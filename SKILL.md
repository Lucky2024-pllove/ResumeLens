# 智能简历初审评估（飞书 CLI 落地版）

## Skill 定位

本 Skill 用于在招聘初筛场景中，对简历做结构化提取与多维评估，输出可直接用于 HR 决策和面试准备的结果。**当用户要求写入飞书（Base 或云文档）时，必须通过 `lark-cli` 执行写入，不得仅用自然语言假装已写入。**

执行前请先阅读飞书 CLI 共享规则（认证、`--as user`/`bot`、scope、`auth login`）。资料来源二选一（内容应对齐）：

- **本仓库内副本（优先）**：[`references/lark-shared/SKILL.md`](./references/lark-shared/SKILL.md)；写入 Base/云文档时另读 [`references/lark-base/SKILL.md`](./references/lark-base/SKILL.md)、[`references/lark-doc/SKILL.md`](./references/lark-doc/SKILL.md)；解析 Wiki 链接时读 [`references/lark-wiki/SKILL.md`](./references/lark-wiki/SKILL.md)。
- **随 CLI 分发的技能包**：`npx skills add larksuite/cli` 等安装方式附带的 **`lark-shared`** 等；或运行 `lark-cli` 时错误提示中的 `console_url` / `hint`。

能力范围：
- 单份简历深度审核（信息提取 + 匹配评分 + 风险识别）
- 批量审核、排序、筛选、Top 候选人推荐
- 多候选人横向对比（差异点、推荐顺序）
- 针对性面试问题生成（优势深化 + 风险验证）
- 薪酬建议与性价比评估

## 使用触发词（建议）

- “帮我初筛这份简历”
- “对这几份简历做排序”
- “给我面试问题”
- “给这个候选人做薪资建议”
- “做候选人对比报告”

## 输入要求

最少输入：
1. 简历文本/文件（单份或多份）
2. 岗位要求（学历、技能、经验、软技能、其他条件）

可选输入（用于飞书落地，**走 lark-cli**）：
3. 输出目标：`chat` / `base` / `doc`（默认 `chat`）
4. Base 目标：`base_token`、`table_id`（若缺失则用 `+base-create` / `+table-create` 创建）
5. Doc 目标：`doc_id` 或文档 URL（若缺失则用 `docs +create` 新建；Wiki 链接需先解析真实 `obj_token`）
6. 写入身份：默认 `--as user`（权限不足时按 lark-shared：补 scope 并 `lark-cli auth login --scope "..."`）

岗位要求建议结构：
- 岗位名称、级别、地点
- 最低学历、专业要求、学历权重
- 核心技能（3-5个，含熟练度和权重）
- 加分技能（2-3个）
- 最低工作年限、行业经验要求、项目规模要求
- 其他关键标准（年龄、地点、到岗状态等）
- 评估权重（学历/能力/经验/软技能/成长等）

若输入缺失：
- 不得编造，明确标注“待补充”
- 继续评估时，对缺失字段写“未明确”

## 标准工作流

### A. 单份简历深度审核（默认）

1) **读取岗位要求**
- 明确硬性门槛（学历、年限、关键技能）
- 明确评分权重

2) **提取候选人信息**
- 姓名、性别、教育经历、最高学历
- 手机、邮箱、年龄
- 工作年限、工作经历
- 意向城市、在职状态

3) **多维评分（1-10分）**

基础维度：
- 学历匹配度
- 能力匹配度
- 经验匹配度

扩展维度：
- 软技能匹配度
- 成长潜力
- 稳定性
- 职业匹配度

真实性维度：
- 时间线一致性
- 项目经历合理性
- 技能熟练度与项目复杂度一致性
- 包装过度风险与信息完整性

4) **综合得分**
- 依据岗位权重做加权求和
- 结合稳定性、职业匹配度、真实性做微调

5) **候选人标签化（3-8个）**
- 能力型：技术专家型/全能型/管理潜力型/快速学习者等
- 技能型：核心栈、稀缺技能、技术深度标签
- 经验型：行业、项目类型、团队规模、职级
- 稳定性：高稳定性/中等稳定性/流动性较大
- 匹配度：紧急可用/长期培养/储备人才/候选备选/需谨慎考虑

6) **生成面试问题**
- 优势深化题：验证亮点深度
- 风险防御题：验证疑点真实性
- 能力验证题：实操/系统设计/边界处理
- 软技能题：沟通协作、抗压、冲突处理

7) **生成薪酬建议**
- 结合学历、经验、技能、项目、公司背景、稳定性
- 给出薪资范围、稀缺溢价、性价比、谈判建议

8) **按指定格式输出**
- 标准 JSON / 简报 / 详细报告 / 批量对比 / HR 面试包

### B. 批量简历处理

1. 逐份执行 A 流程  
2. 按综合得分排序（支持维度排序）  
3. 做硬性条件过滤与阈值筛选  
4. 输出横向对比与 Top 推荐

### C. 多简历对比分析（2-10份）

- 横向对比：学历、能力、经验、软技能、风险、标签
- 输出：对比表 + 差异分析 + 推荐顺序 + 每人独特价值

### D. 输出写入飞书多维表格（Base，可选，**必须用 lark-cli**）

触发条件：
- 用户明确要求“写入飞书多维表格 / Base / bitable”

执行流程（**顺序不得跳步**；写入前向用户确认）：
1. **身份与环境**：默认 `lark-cli ... --as user`；首次使用需 `lark-cli config init`，并按需 `lark-cli auth login --scope "<缺失 scope>"`。  
2. **定位 Base**：若用户无 `base_token`，创建 Base 并回传链接与 token：  
   `lark-cli base +base-create --name "<岗位>-简历初审-<YYYYMMDD>"`  
   可选：`--folder-token <fldcn...>`、`--time-zone Asia/Shanghai`  
3. **定位数据表**：若无 `table_id`，在目标 Base 中建表（字段与下文「飞书落地模板 A」对齐，类型需符合 Base 字段规范）：  
   `lark-cli base +table-create --base-token <bascn...> --name "候选人评估" --fields '[{"name":"姓名","type":"text"},...]'`  
   若表已存在：先 `lark-cli base +table-list --base-token <bascn...>`，再 `lark-cli base +field-list --base-token <bascn...> --table-id <tbl...>` 核对字段名。  
4. **组装记录值**：将「标准 JSON」中的维度映射到表字段；多行文本、标签等类型须按 `lark-cli` Base 记录值规范填写（见本仓库 [`references/lark-base/references/lark-base-shortcut-record-value.md`](./references/lark-base/references/lark-base-shortcut-record-value.md)），禁止凭感觉拼结构。  
5. **写入**：单条可用原生 API 风格创建；批量优先（单次 ≤200 行）：  
   `lark-cli base +record-batch-create --base-token <bascn...> --table-id <tbl...> --json '{"fields":["姓名","综合得分",...],"rows":[["张三",8.2,...],...]}'`  
   超大批次分批调用。  
6. **回执**：返回 CLI 原始结果中的成功条数、失败原因、`base.url` 或用户可打开的 Base 链接；不得虚构。

### E. 输出写入飞书云文档（docs，可选，**必须用 lark-cli**）

触发条件：
- 用户明确要求“写入飞书文档 / 云文档 / docx”

**Wiki 链接**：若用户给的是 `/wiki/wikcn...`，必须先解析真实文档 token：  
`lark-cli wiki spaces get_node --params '{"token":"<wiki_token>"}'`  
再根据 `obj_type` 选择后续 API；云文档正文写入走 `lark-cli docs`（`obj_type` 为 `docx` 时使用返回的 `obj_token` 作为 `--doc`）。

执行流程：
1. **新建文档**（无现成 `doc_id` 时）：用 Lark-flavored Markdown 生成报告正文，例如：  
   `lark-cli docs +create --title "[岗位]简历初审-<YYYYMMDD>" --markdown $'# 报告\n\n...'`  
   可选父位置：`--folder-token`、`--wiki-node`、`--wiki-space`（与 lark-doc `+create` 说明一致）。从返回 JSON 取 `doc_id`、`doc_url` 交给用户。  
2. **更新已有文档**：优先局部更新，避免误删富文本：  
   `lark-cli docs +update --doc "<doc_id_or_url>" --mode append --markdown $'## 新增候选人\n\n...'`  
   慎用 `--mode overwrite`。  
3. **回执**：返回 `doc_url`（注意 Wiki 场景下可能是 `/wiki/...` 链接）、写入模式与失败时的 CLI 错误信息。

## 评分标准（核心）

### 1) 学历匹配度
- 9-10：远超要求，专业高度对口，院校优势明显
- 7-8：达到要求，专业对口或相关
- 5-6：达到要求但优势一般
- 3-4：略低要求，有一定补偿项
- 1-2：明显不达标

### 2) 能力匹配度
- 看核心技能覆盖度、深度、广度与项目实证
- “精通”需有足够项目证据支撑

### 3) 经验匹配度
- 看年限、行业相关性、岗位相关性、项目复杂度、角色深度
- 重点关注最近 2-3 年经历

### 4) 软技能匹配度
- 沟通表达、协作、领导力、学习能力、问题解决

### 5) 成长潜力
- 技术演进速度、职责升级轨迹、项目难度递进

### 6) 稳定性
- 平均在职时长、跳槽频率、职业连贯性

### 7) 职业匹配度
- 职业轨迹与目标岗位发展路径是否一致

### 8) 真实性评分
- 时间线冲突、项目与角色不匹配、技能夸大、关键字段缺失

## 综合得分建议算法

1. 设定岗位权重（示例：学历20%/能力40%/经验40%）  
2. `综合得分 = Σ(维度得分 × 权重)`  
3. 调整规则（示例）：
- 稳定性 < 5：-1
- 职业匹配度 < 5：-1
- 真实性评分 < 5：-2（并标记高风险）
- 成长潜力 > 8：+0.5

## 风险识别清单

重点检测：
- 时间线冲突（多段全职重叠、教育与工作矛盾）
- 空白期过长且无解释
- 工作年限与年龄/学历明显矛盾
- 项目规模与角色描述不匹配
- 技术栈与时间线不合理
- 技能“精通”堆砌且无项目支撑
- 量化成果无数据对比或来源
- 关键信息缺失（时间、公司、项目基础信息）

风险等级：
- 低风险：8-10
- 中风险：5-7
- 高风险：1-4

## 面试问题生成规则

问题设计原则：
- 针对性：围绕简历亮点与疑点定制
- 层次性：由浅入深，先事实再原理再场景
- 证据性：要求案例、数据、结果闭环
- 平衡性：技术与软技能并重

建议题量：
- 初级岗：10-15
- 中级岗：15-22
- 高级岗：22-32

## 薪酬建议规则（参考框架）

考虑因素：
- 学历（院校与层次）
- 年限与岗位相关性
- 核心技能稀缺度
- 项目规模与复杂度
- 公司背景
- 稳定性折扣
- 城市系数（相对一线基准）

输出包含：
- 月薪范围
- 年薪范围
- 稀缺技能溢价说明
- 性价比等级
- 谈判空间与谈判重点

## 飞书落地模板 A：Base 字段设计（推荐）

用途：
- 用于后续筛选、排序、统计、仪表盘展示

字段建议（最小可用）：
- 候选人基础：`姓名`、`性别`、`最高学历`、`年龄`、`工作年限`、`意向城市`、`在职状态`
- 联系方式：`手机号`、`邮箱`
- 评分字段：`学历匹配度`、`能力匹配度`、`经验匹配度`、`软技能匹配度`、`成长潜力`、`稳定性`、`职业匹配度`、`真实性评分`、`综合得分`
- 结论字段：`候选人标签`（多值）、`风险提示`（长文本）、`面试建议`（长文本）、`薪酬建议`（长文本）、`核心评价`（长文本）
- 管理字段：`应聘岗位`、`评估时间`、`评估批次`

推荐映射：
- 数值评分统一用数字字段（1-10）
- 标签可用多选或文本数组（按表能力选）
- 风险提示/面试建议/薪酬建议建议用长文本

写入规则：
- 单份审核：写入 1 条记录
- 批量审核：逐条写入并记录失败项
- 字段缺失：写入“未明确”，禁止伪造

## 飞书落地模板 B：Doc 报告模板（推荐）

用途：
- 适合面试官阅读、HRBP 汇报、归档沉淀

文档结构（单份详细版）：
1. 候选人基本信息  
2. 工作经历与项目摘要  
3. 多维评分与依据  
4. 风险识别与验证建议  
5. 面试问题清单（优势/风险/能力/软技能）  
6. 薪酬建议与谈判提示  
7. 核心评价与录用建议

文档结构（批量对比版）：
1. 处理概览（岗位、数量、时间）  
2. 综合排名表  
3. 维度横向对比（学历/能力/经验/风险）  
4. Top 候选人推荐  
5. 最终建议与后续动作

写入规则：
- 先确认文档目标（新建或更新）
- 标题建议：`[岗位名]简历初审报告-YYYYMMDD`
- 失败回退：写入失败时仍在会话返回完整报告正文

## 执行约束

- 禁止编造简历内容或岗位条件
- 信息不确定时，必须标注“未明确/待补充”
- 评分必须给出依据，保持可追溯
- 批量处理需保持口径一致，避免标准漂移
- 真实性评分低时，不得给出“强烈推荐”
- 涉及写入 Base/Doc 时，必须先确认用户写入意图和目标对象
- 写入失败时不得声称成功，需返回失败原因与补救建议

## 飞书 CLI 速查（Agent 执行清单）

以下命令与 `@larksuite/cli` 及本仓库 [`references/`](./references/) 中 `lark-base` / `lark-doc` 技能正文一致；**云文档请使用 `docs` 服务名，不要使用不存在的 `docx` 顶层命令**。

```bash
# 配置与登录（首次 / 缺 scope）
lark-cli config init --new
lark-cli auth login --scope "<console 或错误提示中的 scope>"

# 新建 Base
lark-cli base +base-create --name "简历初审池" --time-zone Asia/Shanghai

# 在 Base 中新建数据表（字段 JSON 与「模板 A」字段名对齐）
lark-cli base +table-create --base-token bascnXXXXXXXX --name "候选人评估" \
  --fields '[{"name":"姓名","type":"text"},{"name":"综合得分","type":"number"},...]'

# 列出表 / 字段（写入前核对字段名）
lark-cli base +table-list --base-token bascnXXXXXXXX
lark-cli base +field-list --base-token bascnXXXXXXXX --table-id tblXXXXXXXX

# 批量写入评估结果（每批最多 200 行）
lark-cli base +record-batch-create \
  --base-token bascnXXXXXXXX \
  --table-id tblXXXXXXXX \
  --json '{"fields":["姓名","综合得分"],"rows":[["李四",7.5]]}'

# 新建云文档报告
lark-cli docs +create --title "前端岗-简历初审-20260421" --markdown "## 摘要\n\n..."

# 向已有文档追加一节
lark-cli docs +update --doc doxcnXXXXXXXX --mode append --markdown "## 候选人 A\n\n..."
```

**本地只聊天、不写飞书**：可不调用上述命令，但仍建议本机安装 `lark-cli` 以便用户随时切换 `base`/`doc` 输出。
